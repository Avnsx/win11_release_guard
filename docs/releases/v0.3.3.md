## Source Evidence Corrections

`latest_build` is the Microsoft Release Health Current Versions value.
`latest_observed_build` is the newest official observed public Microsoft build
from supported sources and can be newer. `required_baseline_build` is the
compliance floor selected by signed baseline rules. Atom-linked Support
evidence can advance latest-observed context, but latest-observed evidence alone
does not raise the required baseline. When Release Health catches up and the
baseline rules select that same build, all three fields can legitimately match.

Source Diagnostic IDs stay unique when one Atom entry produces multiple
release/build events. The canonical broad-target warning can keep the public
Atom-form ID, for example
`wrg-source-diagnostic-v1:uuid:07747009-7264-44f2-86c2-1c3e09919af3;id=968480`.
Sibling events use deterministic hash-form IDs while preserving
`atom_entry_id`, `atom_support_article_id`, support URL, source URL, and
article-id tag metadata.

Atom is discovery for Support article hrefs, not a fallback KB resolver. The
generator prefers safe Atom `alternate` links to `https://support.microsoft.com`
article paths, ignores `self`/feed/API/search/download/static/non-support
links, strips tracking query strings and fragments from otherwise safe URLs,
accepts no port or explicit `:443`, rejects unsafe ports, userinfo, traversal,
overlong paths, and unsafe hosts, and does not synthesize `/help/<KB>` URLs when
Atom lacks a usable support href.

Support article facts are trusted only after the selected article URL, KB,
expected build, and parseable applicability are compatible with the Atom record.
The bounded `Applies to` extractor supports compact paragraph, heading/list, and
heading/paragraph shapes, records `applies_to_releases` when parseable, and
does not swallow following sections such as prerequisites or known issues.
Mismatched articles remain visible as compact Source Diagnostics validation
metadata, but their KB/title/build facts and Support-derived security wording
are not used for administrator summaries, `Security patch` tags, or security
classification. Degraded but compatible articles keep summaries grounded in
Atom KB/build/release facts and expose the degradation reason.

MSRC CVRF enrichment requires exact KB-token matching. `KB5094126` and bare
`5094126` match as standalone KB tokens; larger tokens such as `KB50941260`,
`15094126`, and `5094126a` do not. Malformed or unavailable CVRF data remains
unknown/unavailable instead of silently becoming `is_security=false`. A minimal
exact-KB remediation still counts as security evidence even when optional CVE,
severity, or product fields are absent. Dashboard and visible JSON exports
publish the classification and evidence source, not CVE lists or counts.

The dashboard can render a blue/white baseline-update notice when a real
Release Health B-release required baseline catches up to the broad target's
latest observed build. It is active only for the 21-day source-derived official
baseline date window, labels date-only Release Health precision explicitly, and
uses deterministic local summarization from Release Health, Atom, validated
Support, and exact MSRC facts. It is dashboard-only: no signed policy verdict,
baseline selection, Source Diagnostics issue sync, runtime client behavior, or
`/api/v1` alias changes are made by the notice.

## Generated Output Regression Coverage

The release adds generated-output tests that inspect the produced policy JSON,
manifest JSON, `/api/v1` aliases, dashboard HTML, and visible Source
Diagnostics copy/export data. The tests cover:

- KB5094126 latest-observed behavior when Release Health has not caught up.
- The caught-up case where `latest_build`, `latest_observed_build`, and
  `required_baseline_build` all equal `26200.8655`.
- Unique diagnostic IDs in policy JSON, rendered `data-diagnostic-id`
  attributes, and copied visible JSON.
- Support article mismatch/degraded states and no raw Support article HTML/text
  leakage.
- MSRC unavailable/malformed status remaining visible as unknown/unavailable.
- Baseline-update notice payload and dashboard rendering order when a real
  B-release baseline has caught up.
- Canonical and `/api/v1` policy/manifest output equivalence.

## Handoff And Archive Rules

`.tmp/prompt-chain/*.patch` files are local hints only. A task is implemented
only when the intended behavior exists in tracked files, tests pass, required
documentation is updated, and logical commits exist. Raw worktree ZIPs are not
release artifacts because they can include `.git/`, `.tmp/`, `site/`, `dist/`,
pycache, package metadata, and private signing-key scratch files.

Clean archives require the `0.3.3` release-note files while keeping historical
`v0.3.2` and `v0.3.1` material available for generated Pages changelog,
release-history, SEO, and auditability routes.

| Distribution check | `python -m twine check dist/*` |
| Publish workflow | `.github/workflows/pypi-publish.yml` |
| Publish method | PyPI Trusted Publishing / GitHub OIDC, no PyPI API token |

If the PyPI project is absent before first publish, configure a Pending Trusted
Publisher first. Pending Trusted Publisher setup does not reserve the package name; if the
name is already controlled by another owner, stop and report instead of
publishing. Do not add PyPI API tokens, Twine passwords, usernames, or
credentialed repository URLs.

## Signed Policy Note

This local version bump does not regenerate the signed bundled production
policy or detached signature. Production policy signing requires the real
private key in the existing secure GitHub Actions workflow. If that key is not
available locally, leave bundled signed policy artifacts unchanged and let the
release/publish workflow produce fresh signed output.

## Unchanged Verdict Boundaries

- Signed public policy JSON plus detached signature remains the compliance verdict authority.
- Local Windows labels, WUA, Panther/setup logs, DISM, Event Logs, and Source Diagnostics remain diagnostic evidence.
- 25H2 remains the existing-device broad-fleet target.
- 26H1 remains excluded for existing-device targeting.
- `/api/v1` policy, signature, and manifest aliases stay backward-compatible.
- Notice Source Diagnostic events remain dashboard-only and are not GitHub-Issue-syncable.
- The baseline-update notice is informational dashboard output only.
